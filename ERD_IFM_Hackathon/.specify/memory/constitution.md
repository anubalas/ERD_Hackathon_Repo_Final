# IFM — Infant Food Manufacturing AI-Powered GMP-Compliant Operations Constitution

> *Powering safer beginnings with AI — where every batch protects a life.*

---

## Core Principles

### I. Zero Silent Failures

Every failure in the sensor telemetry pipeline MUST be logged, alerted, and surfaced to an
operator. Swallowed exceptions on the ingestion path are a critical safety violation.

- Every `POST /telemetry` error MUST emit a structured log entry and trigger an `Alert` record
  in the audit database.
- Redis publish failures MUST be caught, logged, and retried or dead-lettered — never silently
  dropped.
- `IsolationForest` scoring errors MUST halt anomaly processing for that reading and write a
  `PIPELINE_ERROR` alert, not produce a silent default score.
- Health-check endpoints MUST reflect actual downstream health (Redis, ChromaDB, SQLite). A
  healthy response from an unavailable dependency is forbidden.
- `try/except Exception: pass` (bare suppress) is banned on any code path that handles sensor
  data, alert creation, or audit writes.

**Rationale**: Infant food manufacturing tolerates no undetected deviations. A silent sensor
failure is indistinguishable from normal operation and is therefore maximally dangerous. If the
system does not loudly fail, an operator has no basis to act.

---

### II. Immutable Audit Log

The SQLite audit database is append-only. No `UPDATE` or `DELETE` statement is ever executed
on `TelemetryLog`, `Alert`, `AgentRun`, or `BatchRecord` tables — at any layer of the stack.

- All ORM models for audit tables MUST omit update/delete methods in `src/db/crud.py`.
- Schema migrations MUST add columns only — column removal or type changes on audit tables are
  forbidden without a new versioned table and a migration-and-preserve strategy.
- Every audit record carries an ISO 8601 `created_at` timestamp set at insert time. No field
  of a written record is ever overwritten.
- Corrections to erroneous records MUST be made via a new `CORRECTION` record that references
  the original record's `id` — never by modifying the original row.
- Integration tests MUST assert that an update operation on any audit table raises an exception
  or is explicitly blocked by the ORM/DB layer.

**Rationale**: Regulatory inspections under FDA 21 CFR Part 11, ISO 22000, and Codex
Alimentarius HACCP require tamper-evident, time-ordered audit trails. Mutability would
invalidate the entire audit chain and expose the organisation to non-conformance findings.

---

### III. AI Citation Mandate (NON-NEGOTIABLE)

Every response generated by the LangChain/Claude AI agent MUST cite the specific SOP/GMP
document, section, and clause it draws from. Uncited guidance is rejected before it reaches
the operator dashboard.

- The agent system prompt in `src/agent/prompts.py` MUST instruct the model to include a
  `[Source: <doc_name>, §<section>.<clause>]` inline citation for every factual or regulatory
  claim.
- If ChromaDB RAG returns no relevant SOP/GMP chunk for a given anomaly, the agent MUST
  respond with `NO_CITATION_AVAILABLE` and set `requires_human_review = True` rather than
  generate ungrounded guidance.
- Citation format is validated in `src/agent/agent.py` before the response is persisted to
  `AgentRun`. Responses that fail citation validation are logged as `CITATION_VIOLATION` and
  never displayed as authoritative guidance.
- Every response MUST include a numeric confidence value in `[0.0, 1.0]` alongside the
  citation. This value is stored in `AgentRun.confidence_score`.

**Rationale**: GMP regulations require documented, traceable decision support. AI outputs
without citation cannot be defended during regulatory audits and may constitute
non-conformances under 21 CFR Part 110 or Codex Alimentarius CAC/RCP 1-1969 §5 (HACCP).
An agent that speculates without evidence is more dangerous than no agent at all.

---

### IV. Anomaly Model Immutability

The `IsolationForest` baseline model is trained offline on clean batch data and loaded
read-only at runtime. It MUST NOT auto-update, re-fit, or accept new training data from the
live sensor stream under any circumstances.

- The model `.pkl` artifact in `src/detection/models/` is written only by the offline training
  script (`python -m src.detection.anomaly --fit`). No API route or background task may invoke
  `fit()` or `partial_fit()` on any in-memory model instance.
- The runtime anomaly scorer loads the model via `joblib.load()` with no `fit`/`partial_fit`
  calls reachable from the FastAPI process.
- Model version string and training timestamp are captured at training time and stored in the
  model artifact. These are logged at startup and recorded in `BatchRecord.model_version` for
  every scored batch.
- Any request to trigger an online model update via API or dashboard MUST return HTTP 405 and
  emit a `POLICY_VIOLATION` alert to the audit log.
- Deploying a retrained model requires: (1) offline training run, (2) performance validation
  report, (3) QA sign-off, (4) operator acknowledgement logged as a `MODEL_DEPLOY` event.

**Rationale**: Online learning from potentially contaminated or adversarial live data can
silently degrade anomaly detection accuracy. Once a model learns to treat a contaminated CCP
condition as "normal", it actively hides the deviations it was built to detect. Offline
retraining with human sign-off is the only safe model lifecycle pattern for this domain.

---

### V. On-Premise Data Sovereignty

All persistent data — sensor telemetry, audit logs, vector embeddings, and model artifacts —
MUST reside on on-premise infrastructure. No production data leaves the factory network
boundary except the Anthropic Claude API inference call.

- SQLite database files are stored on local disk only. No S3, Azure Blob, Google Cloud
  Storage, or any remote object store integration is permitted in production code.
- ChromaDB MUST be configured with a local `CHROMA_PERSIST_DIR` path. No hosted ChromaDB
  cloud tenants.
- The Anthropic Claude API call (`src/agent/agent.py`) is the only permitted external network
  egress from the production system. Anomaly event context sent to Claude MUST NOT include
  personally identifiable information, batch customer data, or facility-identifying metadata
  beyond what is necessary for GMP remediation reasoning.
- Log files and backup archives remain within the facility's air-gapped network perimeter.
- Any PR introducing cloud export, remote logging, or telemetry forwarding is rejected without
  further review.

**Rationale**: Factory networks handling infant food manufacturing operate under strict data
residency and food safety regulations. Customer and batch data must not be exposed to
third-party cloud providers. The air-gapped factory assumption protects against remote data
exfiltration and ensures regulatory data residency compliance.

---

### VI. Human-in-the-Loop Oversight

When the AI agent's confidence in an anomaly diagnosis or remediation recommendation falls
below the configured threshold, a human review flag MUST be set and the operator MUST be
notified before any recommended action is taken.

- Confidence threshold is controlled by env var `AGENT_HUMAN_REVIEW_THRESHOLD` (default: 0.75,
  range: 0.0–1.0). Any `AgentRun` with `confidence_score < threshold` MUST set
  `requires_human_review = True` before persisting.
- Novel anomaly types — those where ChromaDB returns no matching SOP/GMP chunk — automatically
  set `requires_human_review = True` regardless of confidence score.
- The Streamlit dashboard MUST visually distinguish flagged responses (amber banner +
  "REQUIRES HUMAN REVIEW" label) from confirmed high-confidence responses (green indicator).
- No automated remediation action (batch hold, CCP adjustment command, escalation alert) may
  be triggered without explicit operator acknowledgement on a flagged `AgentRun`.
- All operator acknowledgements are recorded as `HUMAN_ACK` events in the audit log, capturing
  operator ID, decision (accept/override/escalate), and ISO 8601 timestamp.

**Rationale**: Even state-of-the-art language models make mistakes — particularly on novel
anomaly patterns not well-represented in the SOP corpus. In infant food manufacturing, an
incorrect remediation recommendation acted upon without human review could introduce a product
safety failure. Human oversight is both a GMP requirement and a fundamental system safety
property, not an optional user experience layer.

---

### VII. Safety-First Mindset

Every engineering decision — architecture choices, code review acceptance criteria, feature
scope, and performance trade-offs — is evaluated first through the lens of infant safety, not
development velocity.

- Complexity for its own sake is forbidden. Every non-trivial abstraction MUST have a
  documented safety or compliance justification in the corresponding `PLAN.md` Complexity
  Tracking table.
- `async` is the default for all FastAPI routes and Redis subscribers. Blocking the event loop
  delays anomaly detection and is classified as a safety degradation.
- `claude-sonnet-4-6` is the minimum model tier for all GMP agent reasoning tasks. Downgrading
  to Haiku for any decision that reaches the operator or audit log is prohibited.
- Prompt caching (`cache_control: {"type": "ephemeral"}`) MUST be enabled on the system prompt
  block and the SOP context block in every Claude API call to minimise latency on time-critical
  anomaly response paths.
- A degraded-but-operational stance is preferred over a clean-but-down stance: the telemetry
  ingestion and audit log pipelines MUST continue to function even when the AI agent is
  unavailable (circuit-breaker pattern, not cascading failure).

**Rationale**: The system's end consumers — infants — cannot report adverse outcomes, escalate
concerns, or self-advocate. The system must therefore hold itself to the highest standard of
correctness, resilience, and explainability, treating each engineering choice as potentially
life-adjacent.

## GMP Compliance & Regulatory Constraints

This section defines the non-negotiable regulatory anchors that govern all feature scopes,
acceptance criteria, and test plans in this project.

**Applicable Standards:**

| Standard | Scope |
|----------|-------|
| FDA 21 CFR Part 110 | Current Good Manufacturing Practice — Human Food |
| Codex Alimentarius CAC/RCP 1-1969 (Rev. 4-2003) | General Principles of Food Hygiene — HACCP |
| ISO 22000:2018 | Food Safety Management Systems |
| FDA 21 CFR Part 11 | Electronic Records; Electronic Signatures (audit log) |

**Critical Control Points (CCPs):**

All sensor streams monitoring CCPs — temperature, pH, humidity, flow rate, ingredient ratios —
MUST be treated as Principle I-protected. There is no category of CCP telemetry for which
silent failure is acceptable. Every CCP sensor type is listed in the `SensorReading` schema
with an explicit deviation threshold field.

**Deviation Handling:**

- Every anomaly triggers an `Alert` record with severity, source CCP, deviation magnitude,
  and ISO 8601 timestamp.
- All alerts MUST be resolvable to a specific batch via `batch_id` foreign key. Orphaned
  alerts (no `batch_id`) are a data integrity violation and MUST be caught at insert time.
- Agent-generated remediation guidance is advisory, not prescriptive. Human decision authority
  is always preserved under Principle VI.

**Audit Readiness SLA:**

The system MUST be capable of producing a complete, timestamped, unmodified audit trail for
any batch within 60 seconds of an inspector request. The Streamlit dashboard export function
and direct SQLite query API support this SLA.

## Development Workflow & Quality Gates

**Spec-Driven Development (Non-Negotiable):**

No code is written before a feature has an approved `SPEC.md` and `PLAN.md`. The SpecKit
workflow is mandatory:

```
/speckit-specify  →  /speckit-clarify  →  /speckit-plan
    →  /speckit-tasks  →  /speckit-implement  →  /speckit-analyze
```

Every implementation task MUST trace back to a spec requirement. Untraced code in a PR is
grounds for rejection.

**Constitution Compliance Gate:**

All pull requests include a Constitution Compliance checklist. Reviewers verify no principle
in this document is violated. Violations require explicit justification in the PR description
and a `CONSTITUTION_EXCEPTION` event in the audit log if the exception reaches production.

**Testing Gates:**

| Gate | Requirement |
|------|------------|
| Unit | MUST cover all safety-critical paths: ingestion errors, audit write failures, scoring errors |
| Integration | MUST assert append-only behavior — no test may pass if it successfully updates an audit record |
| Citation | MUST verify that an `AgentRun` without a valid citation is rejected by the validation layer |
| Model | MUST assert that no API route can trigger a `fit()` call on the anomaly model |

**Model Deployment Gate:**

Anomaly model updates require all four gates in order: (1) offline training on reviewed
baseline data, (2) performance validation report signed off by QA, (3) explicit operator
acknowledgement, (4) `MODEL_DEPLOY` event written to the audit log before the `.pkl` is
swapped in production.

## Governance

This constitution supersedes all other project practices and documentation. Amendments require:

1. **Proposal** — A written rationale explaining why the amendment is needed and which
   principle(s) it affects.
2. **Impact Assessment** — Review of all dependent specs, plans, tasks, and tests for
   compliance with the amended text.
3. **Approval** — Sign-off by the project lead and at least one domain expert (GMP/food
   safety). For MAJOR version bumps, both must confirm in writing.
4. **Version Bump** — Increment per the versioning policy below.
5. **Propagation** — Update all dependent templates under `.specify/templates/` and the agent
   context file under `.specify/extensions/agent-context/` to reflect the change.
6. **Audit Entry** — Record the amendment as a `CONSTITUTION_AMENDMENT` event in the audit log
   with rationale, approvers, affected principles, and new version number.

**Versioning Policy:**

| Bump | Trigger |
|------|---------|
| MAJOR | A principle is removed, merged with another, or its non-negotiable constraints are weakened |
| MINOR | A new principle is added, or an existing one is materially strengthened or expanded |
| PATCH | Wording clarification, typo fix, formatting — no change to any enforceable rule |

**Compliance Reviews:**

This constitution is reviewed at the start of every sprint and after every regulatory
inspection, incident, or near-miss event. The project lead is responsible for keeping it
current and for ensuring all team members have read the current version.

**Runtime Guidance:**

For runtime development guidance, tooling, shell commands, and active implementation plans,
refer to `CLAUDE.md`, `PLAN.md`, and `SPEC.md` at the repository root.
