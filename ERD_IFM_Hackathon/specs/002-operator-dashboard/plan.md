# Implementation Plan: Streamlit Operator Dashboard

**Branch**: `002-operator-dashboard` | **Date**: 2026-06-09 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-operator-dashboard/spec.md`

---

## Summary

Build a single-file Streamlit dashboard (`src/dashboard/app.py`) that reads from the existing
SQLite `telemetry_log` table (written by feature 001) and presents three panels:
a live sensor feed with 5-second auto-refresh, a CCP alerts panel for REJECTED readings,
and a batch audit trail with search. All data access is synchronous and read-only.

---

## Technical Context

**Language/Version**: Python 3.11

**Primary Dependencies**: `streamlit>=1.35`, `streamlit-autorefresh>=0.0.1`, `sqlalchemy>=2.0`, `pandas>=2.0`

**Storage**: SQLite — same `telemetry_log` table as feature 001; read-only via synchronous SQLAlchemy engine.

**Testing**: Manual UI verification + `pytest tests/unit/test_dashboard_queries.py` for query functions

**Target Platform**: Local desktop browser (factory operator workstation, internal network)

**Project Type**: Single-page internal dashboard

**Performance Goals**: Page renders in < 2 seconds; auto-refresh completes within the 5-second cycle

**Constraints**: No authentication; no writes to DB; must run without anomaly detector or AI agent

**Scale/Scope**: Single operator workstation; ~50 rows in live feed; up to ~1000 rows per batch audit query

## Key Design Decisions

### 1. Single-file implementation

All logic lives in `src/dashboard/app.py`. Three helper functions at module level
(`get_live_feed`, `get_alerts`, `get_batch_audit`) each accept a SQLAlchemy `Session`
and return a `pandas.DataFrame`. The Streamlit UI code calls these functions.
This makes query logic independently testable without a running Streamlit server.

### 2. Synchronous SQLAlchemy engine (module-level singleton)

```python
engine = create_engine(f"sqlite:///{SQLITE_DB_PATH}", connect_args={"check_same_thread": False})
SessionLocal = sessionmaker(bind=engine)
```

`check_same_thread=False` is required for SQLite in Streamlit's multi-threaded server.
The engine is created once at import time and reused across all reruns.

### 3. Auto-refresh via `streamlit-autorefresh`

```python
from streamlit_autorefresh import st_autorefresh
st_autorefresh(interval=5000, key="dashboard_refresh")
```

Placed at the top of the script before any data queries. Triggers a full rerun every
5 seconds without blocking the server thread (avoids `time.sleep` thread-blocking issues).

### 4. Search input preserved via `st.session_state`

```python
if "batch_search" not in st.session_state:
    st.session_state["batch_search"] = ""
```

The text input uses `key="batch_search"` which binds it to session state automatically,
preserving the value across auto-refresh reruns (FR-010 compliance).

### 5. Colour coding via pandas Styler

```python
def _colour_row(row):
    if row["status"] == "REJECTED":
        return ["background-color: #ffcccc"] * len(row)
    elif not row["stream_published"]:
        return ["background-color: #ffe5b4"] * len(row)
    return ["background-color: #ccffcc"] * len(row)

st.dataframe(df.style.apply(_colour_row, axis=1), use_container_width=True)
```

### 6. Graceful DB error handling (Constitution Principle I)

All DB queries are wrapped in `try/except sqlalchemy.exc.OperationalError`.
On failure, a red `st.error()` banner is shown. No bare `except: pass` anywhere.
